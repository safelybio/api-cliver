# Implementation Details

Technical documentation for the Cliver KYC Verification API.

## Models (OpenRouter)

| Model | ID | Purpose |
|-------|-----|---------|
| Main | `google/gemini-3-pro-preview` | Research prompts with tool calling (verification + work search) |
| Extraction | `google/gemini-3-flash-preview` | Structured data extraction from markdown tables |

The main model handles iterative tool calling (up to 20 iterations). The extraction model runs 3 parallel extractions: evidence table, determinations table, and background work table.

## Response Field Computation

### `decision.status`

```python
if sanctions_criterion == "FLAG":
    return "FLAG"  # Only sanctions screening can cause FLAG
elif any_criterion in ("FLAG", "UNDETERMINED"):
    return "REVIEW"  # Other issues require manual review
else:
    return "PASS"
```

Sanctions is treated specially because it indicates a hard compliance failure. Other criteria (affiliation, institution type, email domain) trigger REVIEW to allow human judgment.

### `decision.flags_count`

Count of criteria with `FLAG` or `UNDETERMINED` status.

### `decision.summary`

Generated by extraction model with prompt `summary.txt`. Must be <25 words. Fallback values if LLM fails:
- PASS: "All verification criteria passed."
- FLAG: "Sanctions screening flagged - requires immediate review."
- REVIEW: "Some criteria require manual review."

### `checks[]`

Merged from two separate extractions:
1. **Evidence table** (what sources say about each criterion)
2. **Determinations table** (FLAG / NO FLAG / UNDETERMINED status)

Each check combines evidence + status for the same criterion.

### `background_work[]`

Only populated if `order_description` is provided in request. Extracted from work prompt output. Relevance scale:
- 5 = Customer authored, same organism as order
- 4 = Customer authored, related organism
- 3 = Customer authored, any organism
- 2 = Institution, same organism
- 1 = Institution, related organism

### `audit.tool_calls[]`

Every tool invocation is recorded with:
- Tool name and query/parameters
- Result count
- Full results with assigned citation IDs (`web1`, `epmc2`, `screen1`, etc.)

IDs are assigned sequentially per tool prefix and persist across both verification and work prompts.

## Tools

| Tool | Provider | Cost | API Key Required |
|------|----------|------|------------------|
| `search_web` | Tavily | Paid (free tier available) | `TAVILY_API_KEY` |
| `search_screening_list` | US Trade.gov | Free | `SCREENING_LIST_API_KEY` |
| `search_epmc` | Europe PMC | Free | None |
| `get_orcid_profile` | ORCID.org | Free | None |
| `search_orcid_works` | ORCID.org | Free | None |

### search_web

Tavily "advanced" search depth, max 10 results, 5 chunks per source. Content truncated to 2000 chars.

### search_screening_list

Queries the US Consolidated Screening List with fuzzy name matching. Accepts array of queries (2-5 words each). Deduplicates results by entity name.

**Endpoint**: `https://data.trade.gov/consolidated_screening_list/v1/search`

### search_epmc

Two modes:
- **lite** (default): 25 results, basic metadata, matching authors only
- **full**: 5 results, complete metadata including abstracts

At least one of: `orcid`, `author`, `affiliation`, `topic` required.

### get_orcid_profile

Returns complete researcher profile: employment, education, keywords, external IDs, and up to 5 recent works.

### search_orcid_works

Searches a researcher's publications by keywords. Case-insensitive matching against title, journal, and publication type.

## Verification Criteria

1. **Customer Institutional Affiliation** - Is the customer currently affiliated with their claimed institution?
2. **Institution Type and Biomedical Focus** - Is it a legitimate research/biomedical organization?
3. **Email Domain Verification** - Does the email domain belong to the institution?
4. **Sanctions and Export Control Screening** - Is the institution on US sanctions lists?

### Source Standards

**Sufficient**: Government registries, institution directories, peer-reviewed publications, patents, regulatory filings, Whois records, official screening lists.

**Insufficient**: Customer-written content, wiki pages, unverified submissions.

**Exception**: Institution's own staff directories are valid for criterion 1.

## Non-Obvious Implementation Choices

### Why two models?

The main model (`gemini-3-pro-preview`) is better at reasoning and tool use but slower/more expensive. The extraction model (`gemini-3-flash-preview`) is faster and cheaper for structured extraction. Running 3 extractions in parallel with the fast model is more efficient than sequential calls to the main model.

### Why parallel extractions?

`asyncio.gather()` runs evidence, determinations, and work extractions simultaneously. This reduces latency since they don't depend on each other.

### Why only sanctions causes FLAG?

Sanctions/export control violations are hard compliance failures with legal implications. Other verification failures (unconfirmed affiliation, email mismatch) are suspicious but may have innocent explanationsâ€”hence REVIEW for human judgment.

### Why store raw outputs?

The `audit.raw` field preserves the LLM's full reasoning for debugging and compliance review. The structured extraction can fail or lose nuance; raw text is the ground truth.

### Why deduplicate screening results but not EPMC?

Multiple sanctions list entries for the same entity name are redundant. Multiple publications by the same author are distinct research outputs worth keeping.

### Tool output formatting

Results are formatted as JSON with an instruction header:
```json
{
  "instruction": "Cite using [id] format (e.g., [web1], [epmc2]).",
  "results": [...],
  "metadata": {...}
}
```

This ensures the model knows how to cite and the IDs match what appears in the final response.
